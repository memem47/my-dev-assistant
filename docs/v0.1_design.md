# v0.1 Design: Minimum CLI Prototype

## 1. Purpose

This document describes the design of the v0.1 minimum CLI prototype.

The purpose of v0.1 is to implement a small command-line tool that converts one task description into basic development artifacts.

This version focuses on simplicity, testability, and easy extension.

---

## 2. Target Requirements

This design is based on:

- `docs/v0.1_requirements.md`

v0.1 shall generate:

- Issue title
- Issue description
- Branch name
- Conventional Commit message
- Pull Request title
- Pull Request description

---

---

## 4. Module Responsibilities

### 4.1 `__main__.py`

Entry point for module execution.

It allows the tool to be executed as:

```bash
python -m my_dev_assistant "Add error handling for invalid RAW image size"
```

Responsibility:

- Call the CLI entry function

---

### 4.2 `cli.py`

Handles command-line input and output.

Responsibilities:

- Read command-line arguments
- Validate that a task description is provided
- Call generation logic
- Print generated artifacts
- Return process exit code

`cli.py` shall not contain generation rules.

---

### 4.3 `generator.py`

Contains rule-based generation logic.

Responsibilities:

- Generate Issue title
- Generate Issue description
- Generate branch name
- Generate commit message
- Generate Pull Request title
- Generate Pull Request description

This module should be easy to unit test.

---

### 4.4 `models.py`

Defines data structures used by the tool.

Responsibilities:

- Define an input model if needed
- Define an output model for generated artifacts

Example:

```python
from dataclasses import dataclass

@dataclass
class GeneratedArtifacts:
    issue_title: str
    issue_description: str
    branch_name: str
    commit_message: str
    pr_title: str
    pr_description: str
```

---

## 5. Data Flow

The basic data flow is:

```txt 
Command-line argument
        ↓
cli.py
        ↓
generator.py
        ↓
GeneratedArtifacts
        ↓
cli.py
        ↓
Console output
```

---

## 6. Generation Design

v0.1 uses simple rule-based generation.

No external AI API is used.

### 6.1 Task Type Detection

The tool shall infer the task type from keywords.

Example rules:

| Keywords | Type |
|---|---|
| add, create, implement | feat |
| fix, bug, error, invalid, fail | fix |
| document, docs, README | docs |
| test, unit test | test |
| refactor, clean | refactor |
| update, chore, config | chore |

If no rule matches, use:

```txt
chore
```

---

### 6.2 Short Description Generation

The short description is used for branch names and summaries.

Rules:

- Convert text to lowercase
- Remove unnecessary symbols
- Replace spaces with hyphens
- Keep only important words
- Limit length if necessary

Example:

```txt
Input:
Add error handling for invalid RAW image size

Short description:
invalid-raw-image-size
```

---

### 6.3 Issue Title

Issue title should be short and readable.

Example:

```txt
Invalid RAW image size error
```

### 6.4 Issue Description

Issue description should explain the task simply.

Example:

```md
## Background

The tool needs to handle invalid RAW image size inputs.

## Task

Add error handling for invalid RAW image size.

## Done Condition

- Invalid RAW image size is detected
- An error message is shown
```

---

### 6.5 Branch Name

Format:

```txt
<type>/<short-description>
```

Example:

```txt
fix/invalid-raw-image-size
```

---

### 6.6 Commit Message

Format:

```txt
<type>: <summary>
```

Example:

```txt
fix: handle invalid RAW image size
```

### 6.7 Pull Request Title

Format:

```txt
<Verb> <summary>
```

Example:

```txt
Fix invalid RAW image size handling
```

---

### 6.8 Pull Request Description

Format:

```md
## Summary

This PR handles invalid RAW image size.

## Changes

- Add validation logic
- Add error message for invalid input
```

---

## 7. Error Handling Design

### 7.1 No Input

When no task description is provided:

```txt
Error: task description is required.
```

Exit code:

```txt
1
```

---

### 7.2 Empty Input

When the task description is empty or whitespace:


```txt
Error: task description is required.
```

Exit code:

```txt
1
```

---

### 7.3 Successful Execution

When generation succeeds:

Exit code:

```txt
0
```

---

## 8. Console Output Format

The console output shall be human-readable and copy-friendly.

Example:

```txt
- Add validation logic
- Add error message for invalid input
```

---

## 9. Test Design

Unit tests shall focus on `generator.py`.

Initial test cases:

| Test case | Expected result |
|---|---|
| Input contains `add` | Type is `feat` |
| Input contains `fix` | Type is `fix` |
| Input contains `docs` | Type is `docs` |
| Input contains `test` | Type is `test` |
| Empty input | Error |
| Valid input | All artifacts are generated |

CLI tests may be added later.

---

## 10. Implementation Policy

v0.1 shall follow these policies:

- Keep implementation small
- Avoid external dependencies
- Avoid AI API integration
- Keep generation rules simple
- Separate CLI and generation logic
- Make generation logic easy to test

---

## 11. Future Extension Points

The design should allow future extensions such as:

- GUI support
- Clipboard copy
- JSON output
- Configuration file
- History saving
- GitHub API integration
- AI API integration

These features are outside the scope of v0.1.
